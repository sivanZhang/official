Problem Guide

Problem Workflow:

      1 Create & change a Problem

          In the host page, click 'Problem', then click the 'New a Problem' link get into the page to create a new Problem.
          New a Problem:
          In the page, first select a Product which this Problem related to and then input the other items as usual.
          The button 'Save as Draft' let you save this Problem as draft which means that this draft Problem will not send to approve and cannot new a Fix based on this Problem. And this Problem is only visible to you.
          The button 'Save to Approve' let you save this Problem as formal Problem which means that this Problem will not be sent to approver and can new a Fix based on this Problem. And this Problem is visible to everyone who has permission to use this system.
          Change a Problem:
          In the host page, click 'Problem' get into the Problem index page.
          In the page, click the Problem title to view the Problem detail and change the Problem. Notes: If a Fix is created based on this Problem, the Product of this Problem cannot be changed. If the Problem has been approved by the approver, the Problem cannot be changed too.

      2 Comment a Problem

          In the host page, click 'Problem' get into the Problem index page, click a certain Problem title to comment
          In this detail page, anyone who has the permission to comment a Problem can comment the Problem at any time.

      3 Approve a Problem

          In the host page, click 'Problem' get into the Problem index page.
          In this index page, the user who has the permission to approve a Problem can see a link 'Start Approval' on the right side of a Problem item. Click this item to approve a Problem
          In approval page, the Opinion cannot be null, input it and click a certain button to finish this job

   

          a> Compared to the former system, the Problem is only related to Product information no Release information related in this system.
          b> Compared to the former system, the relationship between Fix and Problem is n:1, which means that multiple Fixes can be created based on one Problem.
          c> If you want to create a Fix, please create or find the related Problem first, because the Fix is based on a Problem.

Fix Guide

Entire Fix Workflow:

      1 Create a Fix

           In this step, we assume that you have created the Problem. If not, please refer the Problem Guide in this page.
          In the host page, click 'Problem' link get into the Problem index page to select a Problem click the 'New Fix' on the right side of a Problem to create a new Fix.
          In the new Fix page, first select a Release version. After select a Release, you can save this Fix as a draft fix at any time or finish all items in this page and save to review to finish this job.
           'Primary Reviewer' is the one who confirm your Fix. Please input the reviewer's username, which is the former PMFKey, like 'zhaji25'. The system will auto-check the validation of the primary reviewer username.
           'Ticket No.' is from Zendesk platform. After inputting the Ticket #, System will auto-check the validation of your input with the Zendesk, so it will take more seconds than usual...
           'Component' could be more than one. So please click 'Add Component' to add your component into component list. System only check the components in the component list not in the select box.
           'prerequisite fix No.': If you have multiple prerequisite Fix No., separate them with ';', and you can check the validation manually by using button 'validation', it will tell you which Fix No. is invalid.
           'Binary File', 'Original Source File' and 'Modified Source File ': Please upload your different files separately. If you save to review this Fix, all these file lists cannot be null.
           'Save to Review' and 'Save as Draft': 'Save as Draft' let creator save a fix as a draft status, which means that this fix is only visible to the creator. So after creator selected a release, any other fields can be empty when save it as a draft, and the creator can edit it later. 'Save to Review' means that the creator save the fix and let others to review it, so the creator need to completely finish every field in that page.

      2 Change a Fix

          You only can change the Fix which is created by you and the status of the Fix is 'created', 'Draft' or 'not Approved'.
          In the host page, click 'Fix' get into the Fix index page, click Fix title get into the Fix detail page and find the change Fix link to change Fix.

      3 Review and Comment a Problem

          After the Fix has been created, the Primary reviewer can receive an email and a message sent by system to notify to review the Fix. An ordinary reviewer can review and comment.
          In the host page, click 'Fix' to find the Fix and click the title get into the detail page to review or to-do list link directly get into the detail page to review.
          In this detail page, select 'confirmed' and click 'comment' to finish confirmation or don't select 'confirmed' and click 'comment', which means that you reviewed this Fix but not confirmed.
          After the Fix has been created, the Fix can be commented at any time if you have the comment permission
          In the Fix index page, you can click the button 'E-mail' to send this Fix to L2 group to review using email system in this system

      4 Approve a Fix

          After the Fix has been confirmed, the approver can approve the Fix.
          In the host page, click 'Fix' to find the Fix get into the index page or to-do list link directly get into the detail page to review.
          In this index page, the user who has the permission to approve a Fix can see a link 'Start Approval' on the right side of a Fix item. Click this item to approve a Fix.
          In the approve page, input the opinion and click a certain button to finish this job

      5 Publish a Fix to Zendesk

          If this Fix need to be published to Zendesk platform, the creator of this Fix should work with the build team to get the formal binary files ready and upload them into our system.

          5.1 Change the Fix Status

          In the host page, click 'Fix' to find the Fix get into the index page.
          In this index page, the user who has the permission to publish a Fix can see a link 'prepare to Create KB' on the right side of a Fix item. Click this button to notify the creator to work with the build team get the binary files ready.

          5.2 Creator upload the new binary files

          In the host page, click 'Fix' to find the Fix and click the Fix title to get into the detail page or to-do list link directly get into the detail page to upload new binary files.
          In the bottom detail page, the user who has the permission to upload new a Fix can see the upload new binary files section. These new binary files should be confirmed by QA, so you need to input a QA using username, we use the PMFKey now.

          5.3 QA confirm the new binary files

          In the host page, click 'Fix' to find the Fix and click the Fix title to get into the index page or to-do list link directly get into the detail page to confirm new binary files.
          In the bottom detail page, the user who has the permission to upload new a Fix can see the 'confirmed' check box. If confirmed, check the 'confirmed', input comments and click 'comment' button to finish confirmation job.

          5.4 Publish Fix to Zendesk

          After QA confirmed, the Fix is ready to be published to Zendesk (you need permission to finish this job).
          In the host page, click 'Fix' to find the Fix or to-do list link directly get into the index page to publish
          In the index page, the user who has the permission to publish Fix can see the 'Ready to Create KB' button. Click this button to publish the Fix to Zendesk, and this operation will take several seconds...
